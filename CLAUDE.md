## Workspace Structure

Seven crates in `crates/`, plus `shuriken-app` which is the main binary:

| Crate | Responsibility |
|---|---|
| `shuriken-core` | Config, error types, core utilities |
| `shuriken-rfc` | RFC parsers/validators (iCalendar, vCard, WebDAV, CalDAV, CardDAV) — no DB/network |
| `shuriken-db` | Diesel ORM, schema, migrations, pure query composition, DB↔domain mappings |
| `shuriken-service` | Business logic: orchestrates DB, auth (Casbin), and RFC layer |
| `shuriken-app` | HTTP server (Salvo), routing, request/response handling |
| `shuriken-test` | Integration test helpers (not published) |
| `shuriken-caldavtester` | Apple ccs-caldavtester runner for RFC compliance |

## Architecture

### Layering and Side-Effect Isolation

- `shuriken-rfc` must stay pure — no DB or network access.
- `src/component/db/query/` functions must stay pure — return queries/expressions, never execute them.
- HTTP handlers acquire DB connections and Casbin enforcer from globals, then pass them down as parameters.
- Services accept `&mut DbConnection<'_>` and an authorizer parameter — no global access inside service functions.

### Data Model

- **Entity/Instance separation**: `dav_entity` holds canonical content shared across collections; `dav_instance` is a per-collection reference.
- **Component tree**: `dav_component` stores hierarchical iCalendar/vCard structure (VCALENDAR → VEVENT, etc.).
- `dav_tombstone` enables correct sync after deletion.
- Monotonic sync tokens and revisions power efficient client synchronization.

### Authorization

- Casbin policies use **UUID-based paths** (e.g., `/cal/<principal-uuid>/<collection-uuid>/**`), not slug-based paths. Slugs can be renamed; UUIDs cannot.
- Use `RESOLVED_LOCATION` from the depot for authorization checks; only use `PATH_LOCATION` when the slug itself matters (e.g., PUT filename).
- Permission expansion: `{user} ∪ groups(user) ∪ {public}`.

## Database Patterns

- **Schema**: `schema.rs` is auto-generated by Diesel — never edit manually. Run `diesel migration run` to regenerate.
- **Primary keys**: UUID v7 via PostgreSQL's `uuidv7()` — time-ordered and globally unique.
- **No `created_at`**: Use `uuid_extract_timestamp(id)` instead.
- **Auto-updated timestamps**: `SELECT diesel_manage_updated_at('table_name');` in migrations.
- **Soft deletes**: `deleted_at TIMESTAMPTZ` where undo windows are needed.
- **Query composition**: Functions in `query/` return `BoxedQuery` for composability — they never take a connection.

```rust
// Good: composable, pure
fn by_id(id: Uuid) -> BoxedQuery<'static> {
    all().filter(users::id.eq(id)).into_boxed()
}
```

## Async Traits

Public traits cannot use `async fn` directly — return `impl Future` instead:

```rust
pub trait Seeder {
    fn seed(&self, conn: &mut DbConnection<'_>) -> impl Future<Output = anyhow::Result<()>>;
}

impl Seeder for MySeeder {
    async fn seed(&self, conn: &mut DbConnection<'_>) -> anyhow::Result<()> { ... }
}
```

## Code Style

- **Clippy**: pedantic lints are `deny` — must fix. Style/complexity/perf are `warn`.
- **Function length**: aim for <40 lines; <50 maximum. Split into parsing/service/serialization helpers.
- **`#[expect]` over `#[allow]`**: documents intentional deviations, errors if the lint disappears.
- **`#[must_use]`**: add to pure functions and builder methods.
- **Option/Result**: prefer `strip_prefix`/`strip_suffix`, `map_or`, `is_some_and` over manual patterns.
- **String building**: use `write!`/`writeln!` instead of repeated `format!` concatenations.
- **Error handling**: `thiserror` for internal component errors, `anyhow` for application flow and boundaries.
- **DRY**: NEVER duplicate logic or constants. Extract shared code into helpers or modules. Duplication is the root of all evil.
- **Structural types**: Invalid states should be unrepresentable. Use enums, newtypes, and private fields to enforce invariants.
- **Strings**: Raw strings are not a good way to pass data around. Parsing strings should be handled in single locations and the parsed data should be passed around in structured types. Two locations should neve have the same string parsing logic. If you find yourself needing to parse the same string format (or similar ones) in multiple places, extract that logic into a helper function and reuse it.

## Test Infrastructure

Integration tests use helpers from `shuriken-test`:

```rust
#[test_log::test(tokio::test)]
#[ignore = "requires running database"]
async fn my_test() {
    let test_db = TestDb::new().await.unwrap();
    test_db.truncate_all().await.unwrap();          // always clean first

    let principal_id = test_db.seed_principal("user", "/principals/alice/", Some("Alice")).await.unwrap();
    let user_id = test_db.seed_user("Alice", "alice@example.com", principal_id).await.unwrap();
    let collection_id = test_db.seed_collection(principal_id, "calendar", "/calendars/alice/personal/", Some("Personal")).await.unwrap();
}
```

HTTP testing:

```rust
let response = TestRequest::propfind("/api/caldav/collection/")
    .depth("1")
    .xml_body(propfind_allprop())
    .send(service)
    .await;

response
    .assert_status(StatusCode::MULTI_STATUS)
    .assert_header_exists("ETag")
    .assert_body_contains("displayname");
```

Control log output with `RUST_LOG=shuriken=debug`. Logs are suppressed unless the test fails (or `-- --nocapture` is passed).
